# V-Sign AWS c6i.large Deploy Plan

Created: 2026-06-12

Decision locked:
- Server runs 24/7 for about 2 months.
- Budget target: under 160 USD for the 2-month server period.
- Instance: AWS EC2 `c6i.large` in `us-east-1`.
- Architecture: `linux/amd64`.
- Server hosts backend, AI service, and Caddy only.
- Frontend deploys separately on Vercel and calls the backend API domain.

---

## 1. Target Production Shape

```text
Vercel frontend
  -> https://api.<domain>/api/v1/*
  -> Caddy on EC2
  -> Spring backend container
  -> private Docker network
  -> AI container
```

Public internet must not reach the AI container directly.

Backend and AI images are built by GitHub Actions and pushed to GHCR. The EC2 server only pulls images and restarts Docker Compose; it does not build application images.

---

## 2. AWS Server Choice

Use:

```text
EC2 instance: c6i.large
vCPU/RAM: 2 vCPU / 4 GB RAM
OS: Ubuntu 24.04 LTS
Disk: 30 GB gp3
Swap: 4 GB
Region: us-east-1
```

Cost estimate for 24/7 over 60 days:

```text
c6i.large: about 122 USD
30 GB gp3 EBS: about 5 USD
Public IPv4: about 7-8 USD
Buffer for transfer/tax/small variance: about 15-25 USD
Expected total: about 135-155 USD
```

Do not use these for the first deploy:
- ALB
- NAT Gateway
- RDS
- CloudFront in front of the API
- ARM/Graviton instances such as `t4g.medium` or `c7g.large`

Reason: the current Docker images are built as `linux/amd64`, and the Python AI stack is lower risk on x86_64 for the first production deploy.

## 4. Server Env Files

Create `/opt/vsign/.env.prod` on the server. It must not be committed.

Required backend values:

```properties
DB_URL=
DB_USER=
DB_PASSWORD=
DB_SCHEMA=vsign_prod

JWT_SECRET=
JWT_EXPIRATION_MS=86400000
JWT_REFRESH_EXPIRATION_MS=604800000

APP_CORS_ALLOWED_ORIGINS=https://<vercel-domain>
AI_SERVICE_BASE_URL=http://ai:8000
AI_SERVICE_PREDICT_TIMEOUT_MS=10000
APP_PAYMENT_PUBLIC_BASE_URL=https://payments.example.invalid
```

Production database schema policy:

- Local development may use the existing Supabase schema `v-sign_schema` because it already contains dev data.
- Production must use a clean dedicated schema, recommended `vsign_prod`.
- Do not use `public` for production. `public` can contain unrelated Supabase objects or an incorrect Flyway baseline.
- Do not reuse `v-sign_schema` for production. It contains old/dev data and previously applied migration history.
- Before the first production deploy, create the schema in the production database:

```sql
create schema if not exists vsign_prod;
```

Flyway should then run from `V1` against `vsign_prod` during the first deploy.

Create `/opt/vsign/.env.ai.prod` on the server:

```properties
VSIGN_AI_API_VERSION=2.0.0
VSIGN_AI_MODEL_VERSION=branch-bilstm-attention-v2
VSIGN_AI_LABEL_VERSION=mvp-3-units-20260609
VSIGN_AI_ENABLE_LEGACY_FRAME_PREDICT=false
VSIGN_AI_MAX_LANDMARK_FRAMES=120
VSIGN_AI_MAX_JSON_BODY_BYTES=524288
VSIGN_AI_PREDICT_TIMEOUT_SECONDS=10
VSIGN_AI_MAX_CONCURRENT_PREDICTIONS=1
```

Runtime deploy values are generated by GitHub Actions into `/opt/vsign/.env.deploy`:

```properties
GHCR_OWNER=<github-owner-lowercase>
IMAGE_TAG=sha-xxxxxxx
APP_DOMAIN=api.<domain>
```

## 6. GitHub Secrets

Add these secrets to the server repository:

```text
SERVER_IP=<elastic-ip-or-host>
SERVER_USER=<ubuntu-user>
SERVER_SSH_KEY=<private-key-for-server>
GHCR_USERNAME=<github-username>
GHCR_TOKEN=<token-with-package-read-if-private>
APP_DOMAIN=api.<domain>
DEPLOY_PATH=/opt/vsign
```

Notes:
- `GITHUB_TOKEN` is already used by workflow to push GHCR images.
- `GHCR_TOKEN` is used on the server to pull GHCR images.
- If GHCR packages are public, `GHCR_TOKEN` can still be kept for predictable auth.

## 7. CI/CD Flow

On push to `main`:

1. GitHub Actions runs backend targeted tests.
2. GitHub Actions runs AI syntax check.
3. Docker Buildx builds backend image as `linux/amd64`.
4. Docker Buildx builds AI image as `linux/amd64`.
5. Both images are pushed to GHCR:

```text
ghcr.io/<owner>/vsign-backend:sha-xxxxxxx
ghcr.io/<owner>/vsign-backend:latest
ghcr.io/<owner>/vsign-ai:sha-xxxxxxx
ghcr.io/<owner>/vsign-ai:latest
```

6. Deploy job uploads root deploy files to `/opt/vsign`.
7. Server runs:

```bash
docker compose --env-file .env.deploy -f docker-compose.prod.yml pull
docker compose --env-file .env.deploy -f docker-compose.prod.yml up -d --remove-orphans
docker image prune -af --filter "until=168h"
```

## 11. Rollback

Use previous GHCR SHA tag:

```bash
cd /opt/vsign
printf "GHCR_OWNER=<owner>\nIMAGE_TAG=sha-previous\nAPP_DOMAIN=api.<domain>\n" > .env.deploy
docker compose --env-file .env.deploy -f docker-compose.prod.yml pull
docker compose --env-file .env.deploy -f docker-compose.prod.yml up -d --remove-orphans
```

Rollback is valid if:

- [ ] Backend `/api/v1/health` returns healthy.
- [ ] AI internal `/health` returns healthy.
- [ ] Caddy serves HTTPS.
- [ ] Lesson flow still reaches completion.

## 13. Hard Boundaries

Do not do these before the first stable deploy:

- Do not move AI to a public URL.
- Do not expose port `8000`.
- Do not deploy frontend on the EC2 server.
- Do not introduce RDS/ALB/NAT unless the budget is revised.
- Do not switch to ARM until backend and AI images are verified as `linux/arm64`.
- Do not implement production payment gateway in this refactor cycle unless Phase 5 is explicitly reactivated.
